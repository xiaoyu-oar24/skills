# docs-layout-quadrant

> **核心思想**：文档是知识图谱，不是 Git Commit 历史。文档应随代码演进而演进，而非每次修改都新建文件。需求文档（What）→ 设计文档（Why）→ 执行计划 → 进度跟踪 → 使用指南，形成完整的软件工程知识链路。

> **生命周期阶段**：稳定

## 🎯 触发条件

- **TRIGGER when**:
  - 开始规划新功能，需要写需求文档、设计文档、执行计划、进度跟踪或使用指南时
  - 用户询问"文档放哪里"或"docs/ 目录怎么组织"
  - 整理混乱的 `docs/` 目录，将散落的文档归类到五维结构中
  - 检查文档命名是否符合规范（发现日期前缀时强制重构）
  - 向 `AGENTS.md` / `CLAUDE.md` 注入或更新 docs 目录结构说明标记块时
  - 当作为 `xy-feat` 协同开发工作流的指南产出与文档合规整理阶段（阶段 6，含规则文件注入）被调用时
- **SKIP**: 对项目根目录 `README.md`、`CLAUDE.md` 等全局配置文件的归类整理（注意：向 `CLAUDE.md` / `AGENTS.md` 注入或更新 docs 结构标记块不在 SKIP 范围内）

## ⚙️ 依赖与先决条件

- 项目已存在 `docs/` 目录。运行 `ls docs/` 或 `ls -d docs` 确认。
- Git 版本控制（用于追踪文件移动历史）。运行 `git --version` 确认。
- 运行环境假设为类 Unix 终端（示例命令使用 `ls`、`git`）。

## 📖 标准工作流

### 阶段 1：命名规范校验（关键前置）

**使用工具**：文件搜索、命令行/终端、编辑文件

扫描 `docs/` 下所有 `.md` 文件，**执行命名规范校验**：

**领域主体命名法** — 文件名仅描述模块/领域，**禁止任何日期/时间戳前缀**：
- `<Bad>` `2026-07-15-网关代理转发-design.md` — 一次性产物，AI 无法判断是否为最终状态
- `<Good>` `网关代理转发.md` — 随代码演进的活文档

**优先中文命名** — 文件名优先使用简明中文描述（如 `网关代理转发.md`），便于人类快速识别文档主题。英文 kebab-case 作为备选（如 `gateway-proxy-design.md`），适用于技术性强的模块或国际化项目。

发现违规文件时，先列出重命名清单并**获得用户确认**，再使用 `git mv` 执行，并修复所有引用该文件的内部链接。

**同名冲突处理**：当多个日期前缀文件去日期后解析为同一目标文件名时（如 `2026-07-15-网关设计.md` 和 `2026-07-20-网关设计.md` 都想去重为 `网关设计.md`），按以下规则仲裁：

1. **识别冲突**：去日期前缀后，检测是否有 2+ 个文件映射到同一目标名
2. **选主版本**：按日期从文件名中提取并比较，**保留最新日期**的那个作为主版本（使用目标名）
3. **归档旧版**：较旧的版本追加 `-archived` 后缀，移入统一归档目录 `docs/.archive/{象限}/`：
   ```
   docs/specs/网关设计.md                       ← 最新版（2026-07-20 的内容）
   docs/.archive/specs/网关设计-archived.md     ← 旧版（2026-07-15 的内容）
   ```
4. **版本合并提示**：冲突解决后，向用户报告：
   > "发现 N 个同名文档的多个日期版本，已保留最新版 `网关设计.md`，旧版已归档至 `docs/.archive/`。请确认是否需要将旧版中的独有内容合并到新版中。"
5. 若文件名中无日期可比较（如 `网关设计.md` 和 `网关设计-v2.md` 同时存在），优先保留无后缀版本，将 `-v2` 版归档

### 阶段 2：五维归类

**使用工具**：命令行/终端（如 `git mv`）、编辑文件

根据文档类型分配到对应子目录：

| 象限 | 路径 | 定位 | 生命周期 |
| :--- | :--- | :--- | :--- |
| 需求文档 | `docs/reqs/` | 产品需求描述（PRD），记录 **What**——页面功能、交互流程、业务规则，是设计文档的输入 | **中长周期** — 功能验收后冻结，作为业务知识长期保留；若需求被废弃则归档至 `docs/.archive/reqs/` |
| 设计规范 | `docs/specs/` | 技术架构决策（ADR），系统的"宪法"，记录 **Why**——架构选型理由、接口设计、数据模型 | **长周期**，随代码演进持续更新 |
| 执行计划 | `docs/plan/` | 实施计划与任务拆解，开发前的 Draft | **短周期** — Feature 交付后追加 `-archived` 并归档至 `docs/.archive/plan/` |
| 进度跟踪 | `docs/tracking/` | 任务 Checklist | **超短周期** — 任务验收后归档至 `docs/.archive/tracking/` 或删除 |
| 使用指南 | `docs/guide/` | 面向开发者的 How-to 操作手册 | **长周期**，随功能迭代更新 |

**弹性降级决策树**：`reqs/` 和 `tracking/` 为按需维度，非所有任务都需创建。AI 必须按以下决策树判定：

#### `reqs/` 判定决策树

```
是否创建 docs/reqs/<功能名>-req.md？
├─ 用户明确要求编写/整理需求文档？ ───► ✅ 创建
├─ 属于纯技术重构/性能优化/API对齐？ ──► ❌ 跳过，需求背景内联至 specs/
├─ 包含新页面、新业务交互或变更有感UI？──► ✅ 创建
└─ 其他无法裁决场景 ───────────────► 向用户提问确认
```

#### `tracking/` 判定决策树（含 xy-feat 作用域保护）

```
是否创建 docs/tracking/<功能名>.md？
├─ 上下文为 /xy-feat 工作流调起？ ────► ✅ 强制创建（防多 Agent 并发写入冲突）
├─ 使用了 dispatching-parallel-agents？──► ✅ 强制创建
├─ 单线程任务且卡片/步骤 ≤ 3 个？ ────► ❌ 跳过，进度 Checklist 内联至 plan.md 尾部
└─ 其他独立任务场景 ───────────────► 默认创建（宁可多建，不漏追踪）
```

> **核心三支撑**：`specs/` + `plan/` + `guide/` 为任何功能开发的核心必选维度，不受决策树影响。

### 阶段 3：象限生命周期管理

**使用工具**：目录浏览、命令行/终端（如 `git mv`）

对已有文档执行过期清理：

1. **plan/** — 扫描已完成功能的 plan 文档：
   - 已交付 → 追加 `-archived` 后缀，移入 `docs/.archive/plan/`
   - **内联降级处理**：若 plan 文档尾部包含内联的 `## 📋 实施进度 Checklist`，则随 plan 文档整体归档（无需单独拆分）
   - AI 不应再将已完成的 plan 作为系统现状参考源
2. **tracking/** — 扫描已验收的追踪文档：
   - 独立 tracking.md → 按以下决策树处理：
     - 包含设计讨论/决策记录 → 移入 `docs/.archive/tracking/`
     - 纯勾选框清单，无讨论价值 → 直接删除（需先向用户说明并获得确认）
   - **代码库不该存储执行状态**，Git 历史已足够
3. **reqs/** — 扫描需求文档：
   - 功能已验收 → 保持原位（作为业务知识长期保留）
   - 需求已废弃（功能下线或被替代）→ 追加 `-archived` 后缀，移入 `docs/.archive/reqs/`
4. **specs/ & guide/** — 作为长周期文档，保持原位，但检查内容是否过时

### 阶段 4：建立知识索引

**使用工具**：写入文件、读取文件

在 `docs/` 根目录维护 `INDEX.md`（若不存在则创建），作为 AI 和人类的知识入口：

```markdown
# 📚 项目知识索引

> 本文件是 docs/ 目录的导航入口，按业务领域组织。新增文档时必须同步更新此索引。

## [领域名称]
| 文档 | 象限 | 文件 |
| :--- | :--- | :--- |
| 网关代理转发需求 | reqs | [gateway-proxy-req.md](reqs/gateway-proxy-req.md) |
| 网关代理转发设计 | specs | [gateway-proxy-design.md](specs/gateway-proxy-design.md) |
| 网关使用指南 | guide | [gateway-proxy-guide.md](guide/gateway-proxy-guide.md) |

## 📦 归档文档
> 仅列出仍有参考价值的归档文档（如被替代的旧版设计），纯 tracking 清单不列入。
| 文档 | 原象限 | 归档原因 | 文件 |
| :--- | :--- | :--- | :--- |
| 网关设计 v1 | specs | 被 v2 替代 | [.archive/specs/网关设计-archived.md](.archive/specs/网关设计-archived.md) |
| 网关需求 v1 | reqs | 需求废弃 | [.archive/reqs/网关需求-archived.md](.archive/reqs/网关需求-archived.md) |
```

每次新增/删除/重命名文档后，**必须同步更新** `docs/INDEX.md`。归档文档移入 `docs/.archive/` 时，若有参考价值也需在 INDEX 的"📦 归档文档"小节中体现。

### 阶段 5：移动文件与修复链接

**使用工具**：命令行/终端（如 `git mv`）、文本搜索、编辑文件

1. 使用 `git mv <旧路径> <新路径>` 移动文件（保留 Git 历史）；若 `docs/` 未被 Git 跟踪（如被 `.gitignore` 忽略），退化为普通 `mv` 并向用户说明
2. 搜索文档中引用了旧路径的链接，逐个修正为新的相对路径
3. 所有文档间链接必须使用相对路径（如 `../specs/xxx.md`），**禁止**使用绝对路径

### 阶段 6：验证

**使用工具**：文本搜索

搜索：
- 旧路径残留 → 确认所有引用已更新
- 日期前缀残留（`20[0-9]{2}-[0-9]{2}-[0-9]{2}-.*\.md`）→ 确认已全部重命名
- INDEX.md 中的链接 → 确认全部可访问

### 阶段 7：向项目规则文件注入 docs 结构说明

**使用工具**：读取文件、写入文件

检查项目根目录是否存在 `AGENTS.md` 和/或 `CLAUDE.md`，向其中追加标准化的 docs 目录结构说明段落，帮助 AI 助手在后续会话中理解文档组织规范。

**注入策略与授权**：
1. 读取目标文件，按前缀匹配检查是否已包含 `<!-- docs-layout-quadrant:start` 标记（标记尾部带版本号，如 `(v2.5)`）
2. 若已存在 → 替换整个标记块（含同步刷新 `start` 标记行中的版本号后缀为最新 `(vX.Y)`），标记块在文件中的位置保持不动
3. 若不存在 → 在文件末尾追加标准化段落
4. **确认要求**：独立调用本技能时，修改规则文件前需先向用户简要说明要写入/更新的标记块内容；在 `xy-feat` 工作流上下文中，用户调用 `/xy-feat` 视为预授权规则文件标记块的标准化注入与更新

**注入内容模板**：

```markdown
<!-- docs-layout-quadrant:start (v2.5) -->
## 📁 docs/ 目录结构

本项目采用五维文档布局，由 `docs-layout-quadrant` 技能维护。AI 助手在读取/创建/修改文档时请遵循此结构。

| 目录 | 用途 | 存放规则 |
| :--- | :--- | :--- |
| `docs/reqs/` | 产品需求描述（PRD），记录 What——页面功能、交互流程、业务规则 [按需-决策树] | 中长周期，验收后冻结保留；废弃需求归档至 `docs/.archive/reqs/` |
| `docs/specs/` | 技术架构决策（ADR），记录 Why——架构选型、接口设计、数据模型 [核心必选] | 长周期，随代码演进持续更新 |
| `docs/plan/` | 实施计划与任务拆解 [核心必选] | 短周期，交付后归档至 `docs/.archive/plan/` |
| `docs/tracking/` | 任务进度追踪清单 [xy-feat必选/单任务按需] | 超短周期，验收后归档或删除 |
| `docs/guide/` | 面向开发者的 How-to 操作手册 [核心必选] | 长周期，随功能迭代更新 |
| `docs/.archive/` | 统一归档目录 | 历史版本、已完成计划、废弃文档 |
| `docs/INDEX.md` | 知识导航入口 | 新增/删除/重命名文档后必须同步更新 |

### 弹性降级
- `reqs/` 和 `tracking/` 为按需维度：纯技术重构可跳过 reqs/（需求背景内联至 specs/），单线程 ≤3 步任务可跳过 tracking/（进度内联至 plan/）
- **核心三支撑** `specs/` + `plan/` + `guide/` 为任何功能开发必选，不受降级影响

### 命名规范
- **禁止日期前缀**：文件名使用领域主体命名（如 `网关代理转发.md`），不使用 `2026-07-15-xxx.md`
- **优先中文命名**：优先使用简明中文命名文件，便于人类快速识别；英文 kebab-case 作为备选
- 文档间引用使用相对路径（如 `../specs/xxx.md`），禁止绝对路径
- 归档文件统一追加 `-archived` 后缀，移入 `docs/.archive/{象限}/`
<!-- docs-layout-quadrant:end -->
```

**注意事项**：
- 不改动标记块之外的任何已有内容；首次注入时在文件末尾追加，更新时已存在的标记块（连同 `start` 标记行上的版本号）整体替换
- 若目标文件不存在，跳过该文件（不强制创建 AGENTS.md / CLAUDE.md）
- 标记中的版本号取技能 version 的主.次版本号（如 `2.5.1` → `v2.5`），用于后续版本升级时定位和替换，不要手动删除标记注释

## ⛔ 行为护栏

- **🚫 禁止日期前缀 + 优先中文命名**：文件名不得以日期开头，必须使用领域主体命名（如 `网关代理转发.md`）。优先使用简明中文描述，英文 kebab-case 作为备选。发现违规时必须重命名并修复引用。文档是知识图谱，不是流水账。
- **必须使用相对路径**：文档间互相引用时，必须使用相对路径（`./` 或 `../`），禁止使用任何绝对路径
- **禁止散落文档 + 保护入口文件**：`docs/` 根目录下只能保留 `INDEX.md` 等入口文件，所有功能性文档必须归入五个子目录。移动和重命名文档时，确保项目根目录的 `README.md` 不受影响。`AGENTS.md` / `CLAUDE.md` 仅由阶段 7 通过标记块方式修改，其他阶段不得编辑
- **规则文件注入标记不可损坏 + 注入须有授权**：阶段 7 注入的 `<!-- docs-layout-quadrant:start (vX.Y) -->` / `<!-- docs-layout-quadrant:end -->` 标记用于版本升级时定位和替换，任何其他操作不得删除或修改这两个标记之间的内容。修改 `AGENTS.md` / `CLAUDE.md` 前需向用户说明（在 `/xy-feat` 工作流下视作预授权）
- **必须使用 `git mv` 保留 Git 历史**：移动已被 Git 跟踪的文件时必须使用 `git mv` 而非普通 `mv`，以完整保留版本变更历史；未跟踪文件（如 `docs/` 被 `.gitignore` 忽略时）退化为普通 `mv` 并向用户说明
- **删除/批量移动前需确认**：任何删除或批量移动、重命名文件的操作，必须先列出受影响文件清单并获得用户确认后方可执行
- **🚫 禁止空壳文档**：经决策树判定跳过 `reqs/` 或 `tracking/` 时，禁止强行生成空白/空壳 `.md` 文件占位。降级内容必须使用标准内联模板写入宿主文档
- **索引同步不可跳过**：任何文档增删改后，必须同步更新 `docs/INDEX.md`，保持导航网最新

## 📝 模板与范例

### <Bad> 反模式

```markdown
# ❌ 时间戳命名 + 绝对路径 + 散落文档

混乱的文档组织（docs/ 根目录散落文件）：
- /docs/2026-07-15-网关代理转发-design.md
- /docs/2026-07-20-网关代理转发-v2.md   ← 每次修改新建文件
- /docs/跟进度第三期报告.md

绝对路径链接（反模式）：
[详细设计](<绝对路径>/docs/2026-07-15-网关代理转发-design.md)

问题：
1. AI 搜索"网关如何工作"时，无法确定哪个是最新状态
2. 散落的文件让 AI 缺乏知识入口
3. 日期前缀让文档变成一次性产物，写完就再也不维护
```

### <Good> 规范示例

```markdown
# ✅ 领域主体命名 + 五维 + 知识索引

规范的五维组织：
- docs/INDEX.md                        ← 知识导航入口
- docs/reqs/gateway-proxy-req.md       ← 需求文档，验收后冻结保留
- docs/specs/gateway-proxy-design.md   ← 随代码演进的长活文档
- docs/plan/gateway-proxy-plan.md      ← 交付后追加 -archived 并归档至 docs/.archive/plan/
- docs/tracking/gateway-proxy.md       ← 验收后归档或删除
- docs/guide/gateway-proxy-guide.md    ← 面向开发者的 How-to
- docs/.archive/                       ← 统一归档目录

相对路径链接：
[详细设计](../specs/gateway-proxy-design.md)

INDEX.md 导航：
## 网关代理
| 文档 | 象限 | 文件 |
| :--- | :--- | :--- |
| 需求文档 | reqs | [gateway-proxy-req.md](reqs/gateway-proxy-req.md) |
| 设计规范 | specs | [gateway-proxy-design.md](specs/gateway-proxy-design.md) |
| 使用指南 | guide | [gateway-proxy-guide.md](guide/gateway-proxy-guide.md) |
```

### <Good> 内联降级模板（按需维度跳过时使用）

**`reqs/` 内联降级** — 纯技术重构/性能优化/API对齐跳过独立 PRD 时，在 `specs/<功能名>-design.md` 头部写入：

```markdown
# [功能名] 技术架构与设计规范

> **降级说明**：本任务属于纯技术改造/优化，无独立业务 PRD。需求背景已内联在本文档中。

## 1. 需求背景与改造目标
- **当前痛点**：[简述现有架构/代码痛点]
- **改造目标**：[简述改造后达成的技术指标/收益]
- **非目标 (Non-Goals)**：[明确本次不做的范围]
```

**`tracking/` 内联降级** — 单线程轻量任务跳过独立 tracking.md 时，在 `plan/<功能名>-plan.md` 尾部写入：

```markdown
---

## 📋 实施进度 Checklist (内联 Tracking)

> **降级说明**：本任务为单线程轻量任务，进度追踪已内联在 plan 中，不单独生成 tracking.md。

- [ ] **阶段 1：准备与重构**
  - [ ] 任务 1.1: [具体子任务描述] `file_path:line`
  - [ ] 任务 1.2: [具体子任务描述]
- [ ] **阶段 2：核心逻辑实现**
  - [ ] 任务 2.1: [具体子任务描述]
- [ ] **阶段 3：验证与交付**
  - [ ] 任务 3.1: 运行单元测试 / 验证点
```
