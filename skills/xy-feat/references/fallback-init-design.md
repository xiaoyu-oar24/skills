# xy-feat 降级流程：阶段 0（隔离开发）与阶段 1（方案设计）

> 本文件仅在 superpowers 对应技能（`using-git-worktrees` / `brainstorming`）不可用时按需读取，正文为完整内联替代流程。

## 阶段 0 降级：隔离开发（可选）

当用户提到"隔离开发"或当前工作区已有未提交变更时：

```
步骤 1：检测当前环境
  git rev-parse --show-toplevel  → 确认在 git 仓库中
  git branch --show-current      → 确认当前分支

步骤 2：询问用户
  "是否需要在隔离的 git worktree 中进行开发？"

步骤 3：若用户同意
  mkdir -p .worktrees
  git worktree add .worktrees/<功能名> -b feat/<功能名>
  cd .worktrees/<功能名>

步骤 4：验证基线
  npm test / cargo test / pytest
```

若平台已有原生隔离工具，优先使用原生工具。

## 阶段 1 内联流程：方案设计

### 1.1 需求澄清（草案优先，仅在歧义时提问）

- 阅读项目现有文件、文档、最近提交，理解当前上下文。
- 若需求模糊，向用户提问以细化需求，**多选优先于开放式**，提问做到精准不冗余：
  - 目标是什么？成功标准？
  - 有什么约束条件（技术栈、性能、兼容性）？
  - 与现有系统的接口关系？
- 若需求包含多个独立子系统，先协助用户拆分为子项目，单独走 xy-feat 流程。

### 1.2 方案对比

- 针对核心难题，评估 **2-3 种可行方案**，列出各自的优劣势和推荐方案，一并放入 Draft 中供用户审批。

### 1.3 一站式 Draft 呈现与审批

- **拒绝逐节打断**：将方案全貌整理为一份完整的结构化设计 Draft，一次性展示（架构、组件、数据流、错误处理、测试策略）：
  - 简单部分：几句话概括
  - 复杂部分：200-300 字详细说明与代码伪代码
- 仅在遇到**重大底座选型冲突**（如 2-3 种完全对立的技术架构选型）时才单独提问确认，其余情况下直接呈现 Draft 供用户统一审批。
- **设计隔离原则**：每个模块应有明确边界和接口，可独立理解和测试。

### 1.4 撰写设计文档

- 将最终审批通过的设计写入 `docs/specs/<功能名>-design.md`。
- **reqs/ 内联降级要求**：若经决策树判定（如纯技术重构/优化）跳过独立 `docs/reqs/` 需求文档，必须在 `docs/specs/<功能名>-design.md` 的头部包含【`## 1. 需求背景与改造目标`】章节，明确背景与问题定义。
- 文档应包含：需求背景与改造目标（降级时必填）、方案选择理由、架构设计、接口定义、数据模型、测试策略。

### 1.5 设计文档自审

写完后以新视角快速自审：
1. 占位符扫描：是否有 "TBD"/"TODO"/未完成的章节？
2. 内部一致性：不同章节之间是否有矛盾？
3. 范围检查：是否聚焦足够，还是需要进一步拆解？
4. 歧义检查：是否有任何需求可被两种方式解读？

修正发现的问题后，请用户审阅 spec 文件，获得批准后再进入阶段 2。
