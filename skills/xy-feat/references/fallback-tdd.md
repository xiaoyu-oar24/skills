# xy-feat 降级流程：阶段 3（TDD/VDD 执行）

> 本文件仅在 superpowers 对应技能（`test-driven-development` / `systematic-debugging` / `dispatching-parallel-agents` / `subagent-driven-development`）不可用时按需读取，正文为完整内联替代流程。

## 3.1 执行原则与模式判定

在开始编写具体代码前，先判定开发模式：

```
前置判断：当前任务是否属于纯 UI/CSS 样式微调、配置文件 (Dockerfile, CI 脚本) 或无单测框架环境？
├── 否 → 严格执行 TDD Iron Law: NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
│        （先写测试 → 看它失败 → 写最小实现 → 看它通过 → 重构）
└── 是 → 自动切换为 VDD (Validation-Driven Development)
         （RED 阶段运行静态检查 tsc/stylelint 或打印缺失修饰状态断言 → GREEN 阶段写入最小修改通过检查）
```

## 3.2 模式循环流程

### 模式 A：TDD 循环（常规功能开发）

```
RED（红灯）：
  1. 基于 plan 中定义的测试，编写一个最小测试用例
  2. 运行测试 → 必须 FAIL
  3. 确认失败原因是你预期的（如 "function not defined"），而非笔误

GREEN（绿灯）：
  4. 编写最少量的代码使测试通过
  5. 运行测试 → 必须 PASS
  6. 不能添加测试未要求的功能（YAGNI）

REFACTOR（重构）：
  7. 消除重复、改善命名、提取公共逻辑
  8. 保持测试绿灯，不添加新行为
  9. 进入下一个 RED 循环
```

### 模式 B：VDD 循环（UI/CSS 样式/配置文件/无单测环境）

```
RED（红灯）：
  1. 运行对应场景的静态检查工具（如 tsc / stylelint / dockerfilelint / eslint）
  2. 或在控制台/测试断言中确认当前样式/配置处于未满足要求的状态
  3. 确认校验命令输出包含预期的缺失状态或错误

GREEN（绿灯）：
  4. 写入支持该样式/配置的最小修改
  5. 重新运行静态检查或确认状态 → 必须全量 PASS 无错误

REFACTOR（重构）：
  6. 清理冗余属性与硬编码配置，确保遵循项目统一规范
```

## 3.3 任务调度与并发安全

**按以下决策树选择执行方式：**

```
任务卡片中有 2+ 个无共享状态的独立任务？
  ├── 是 → 并行调度（见 3.3.1）
  └── 否 → 单个任务内部步骤多、复杂度高？
            ├── 是 → 子代理独立执行（见 3.3.2）
            └── 否 → 顺序执行（见 3.3.3）
```

### 3.3.1 并行调度（含子代理并发安全禁令）

当多个任务完全独立（编辑不同文件、无共享状态）时：
- 使用子代理调度，在**同一条消息**中同时派发所有子代理。
- 每个子代理获得具体任务范围与上下文，不继承主会话历史。
- **🚨 核心并发安全禁令**：
  - **禁止任何并行子代理直接使用 Edit 修改全局 `docs/tracking/<功能名>.md` 文件**。
  - 子代理只需在其最终返回结果中包含 `Task N Complete (commit sha)`。
  - 由**主编排代理**在接收并收齐所有并行子代理的结果后，集中批量更新 `docs/tracking/<功能名>.md`，彻底消除竞态覆盖风险。

### 3.3.2 子代理独立执行

当单个任务卡片内部步骤多、逻辑复杂时：
- 派发一个子代理独立完成该任务的全部 TDD/VDD 循环。
- 子代理的任务简报与执行汇报**直接在提示词和返回结果中传递，不落盘为文件**。

### 3.3.3 顺序执行

不满足并行或子代理条件时，按 task 列表顺序由主代理逐个执行。

## 3.4 实时追踪（主编排代理负责）

每确认完成一个任务卡片，主编排代理立即编辑 `docs/tracking/<功能名>.md` 中对应条目标记为 `✅`，并记录关键提交 hash：

```markdown
- ✅ [后端] Task 1: 创建 API 路由（commits abc123..def456，测试 5/5 通过）
- ✅ [前端] Task 3: 样式调整（commits def456..ghi789，VDD 静态校验 PASS）
```

## 3.5 遇到 Bug 或测试失败

```
NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
```

**内联自 `systematic-debugging` 的四阶段调试法：**

### Phase 1：根因调查（禁止跳过）
1. **精读错误信息**：不跳过报错或警告，读完整栈追踪。
2. **稳定复现**：确认稳定触发，记录复现步骤。
3. **检查最近变更**：`git diff`、最近提交、新依赖、配置变更。
4. **数据流追踪**：错误值源头向上追溯。
5. **多组件系统**：在组件边界集中插桩定位故障层。

### Phase 2：模式分析
- 在同一代码库中寻找类似的、正常工作的代码。
- 逐行对比差异，不放过任何微小区别。

### Phase 3：假设与验证
1. **形成单一假设**："我认为 X 是根因，因为 Y"。
2. **最小化验证**：做最小改动验证，一次只改一个变量。
3. **根据结果决策**：
   - 验证通过 → 进入 Phase 4
   - 验证失败 → 形成**新的、不同的**假设（不能重复同一假设）

### Phase 4：实施修复
1. 基于根因编写能复现 Bug 的失败测试或静态断言。
2. 实施**单一修复**（仅改根因，不做顺手优化）。
3. 验证：测试通过、无回归、Bug 确实解决。

### 重试策略
- **最多重试 3 次**（必须基于不同假设）。
- 同一假设连续失败 2 次 → 立即暂停并汇报用户。
- 3 次不同假设均失败 → 质疑架构模式，汇报用户讨论重构。

## 3.6 跨任务上下文对齐

在开始实现下一个任务卡片前，重新查看 plan 文档的设计上下文，严防"文档写一套、代码写另一套"。
